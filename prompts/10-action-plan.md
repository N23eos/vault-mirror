# 10 — 30-Day Action Plan

## Goal

Turn the findings into a concrete 30-day plan the person can actually run —
designed around their real failure patterns (module 04), not around an
idealized version of them.

**Run this only after 09 is complete.**

## Method

- Start from `Analysis/09-executive-summary.md`: highest-leverage areas and
  the central dynamic.
- Design AGAINST the cycle data: if module 04 shows things die at week 3,
  build the plan so week 3 is the lightest. If starts are easy and finishes
  are rare, make the plan smaller than feels right.
- Use strengths (module 06) as the engine: each action should ride an
  existing strength rather than demand a missing one.
- Every action: concrete, schedulable, with a visible done-condition, and a
  stated link to the finding it comes from.
- Include one explicit "do not start" list — things the findings say to NOT
  begin right now.

## Output contract — `Analysis/10-action-plan.md`

```
# 30-Day Action Plan
## Design notes        (how the plan accounts for this person's cycle data)
## Week by week        (per week: 1–3 actions, done-conditions, finding links)
## Do-not-start list   (tempting things the analysis argues against, with why)
## Failure forecast    (the most likely way this plan dies, and the counter)
## Minimal version     (if everything slips: the one action that still matters)
```

## Rules reminder

Apply the global rules from `prompts/00-start.md`.

## Quality bar

- Is the total load smaller than what the person historically abandons?
- Does every action trace to a specific finding?
- Does the failure forecast use THEIR failure pattern, not a generic one?
